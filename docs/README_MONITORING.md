# Monitoring Guide

Актуальное руководство по встроенному мониторингу в `VPN Server Manager`.

Дата обновления: `2026-04-06`  
Статус: `актуально для текущего кода`

---

## Что умеет мониторинг

Страница мониторинга собирает данные по SSH и показывает 5 блоков:

1. Сетевой трафик.
2. Статус firewall и listening ports.
3. Системные сервисы.
4. События безопасности.
5. Историю CPU и памяти.

Основные backend-части:

- `app/services/ssh_service.py`
- `app/routes/api.py`
- `templates/monitoring.html`

Основные API:

- `GET /api/monitoring/<server_id>/network-stats`
- `GET /api/monitoring/<server_id>/firewall-stats`
- `GET /api/monitoring/<server_id>/services-stats`
- `GET /api/monitoring/<server_id>/security-events`
- `GET /api/monitoring/<server_id>/metrics-history`
- `GET /api/monitoring/<server_id>/check-tools`

---

## Как это работает

Мониторинг не ставит постоянного агента. Данные собираются по SSH с переиспользованием соединений.

Ключевые особенности:

- используется SSH connection pooling, чтобы не открывать лишние подключения;
- endpoints защищены rate limiting;
- фронтенд обновляет блоки с разной частотой, чтобы не перегружать сервер;
- история CPU/Memory хранится на сервере в `/var/tmp/metrics_history.json`;
- для сетевой статистики за сутки используется `vnstat`, если он установлен.

## Что именно показывает UI

### 1. Сетевой трафик

Источник:

- `/sys/class/net/*/statistics/*` для текущей скорости;
- `vnstat` для суточного трафика.

Показывается:

- текущая загрузка/отдача;
- пики за текущую сессию страницы;
- список мониторимых интерфейсов;
- суммарный трафик за сутки.

### 2. Firewall и порты

Мониторинг теперь смотрит не только на `ufw`.

Логика:

- сначала определяется backend firewall;
- отдельно собираются реальные listening ports;
- для `ufw` дополнительно читаются блокировки из `/var/log/ufw.log`.

Поддерживаемые варианты определения backend:

- `ufw`
- `firewalld`
- `nftables`
- `iptables`
- `none`

Показывается:

- статус firewall (`active` или `inactive`);
- backend firewall;
- listening ports, реально открытые на сервере;
- количество `UFW BLOCK` за 24 часа, если используется `ufw`;
- последний заблокированный IP и порт, если эти данные есть.

Важно:

- поле `open_ports` в UI сейчас означает именно слушающие порты, а не только правила `ufw`;
- это полезнее для диагностики, потому что показывает реальное состояние сервера.

### 3. Сервисы

Проверяются типовые сервисы:

- `nginx`
- `apache2`
- `ssh`
- `sshd`
- `postgresql`
- `mysql`
- `docker`
- `redis-server`

Показывается:

- активен ли сервис;
- включен ли в автозапуск;
- uptime сервиса.

### 4. События безопасности

Источник данных:

- `journalctl --since "24 hours ago"` как основной источник;
- `auth.log` как fallback, если journald недоступен или не дал данных;
- `fail2ban-client` для текущего числа забаненных IP;
- `systemctl --failed` для проблемных unit'ов;
- `journalctl -p err..alert` для количества ошибок уровня `err` и выше;
- `apt list --upgradable` для security updates.

Показывается:

- количество неудачных SSH login попыток за 24 часа;
- топ IP с этими попытками;
- доступные security updates;
- сколько дней прошло с последнего обновления пакетов;
- сколько появилось новых listening ports относительно baseline;
- сколько IP сейчас держит `fail2ban`;
- сколько failed `systemd` unit'ов;
- сколько ошибок в journal за последние 24 часа.

### 5. История CPU и памяти

Показывается история последних точек CPU/Memory.

Особенности:

- максимум хранится 60 точек;
- если на сервере есть `jq`, обновление истории выполняется проще и надежнее;
- без `jq` используется fallback на Python-обработку на стороне приложения.

## Требуемые и полезные утилиты

Мониторинг старается работать даже в минимальной системе, но часть функций становится лучше при наличии дополнительных пакетов.

### Рекомендуется

- `vnstat` - суточный сетевой трафик;
- `jq` - удобное обновление JSON-истории;
- `net-tools` - дополнительный fallback через `netstat`.

### Опционально

- `ufw` - только если вы реально используете его как firewall backend;
- `fail2ban` - полезно для защиты SSH.

---

## Важное про UFW

Мониторинг не требует включенного `ufw`.

Правильная позиция:

- `ufw` не обязателен;
- если он уже используется на сервере, мониторинг покажет его статус;
- если `ufw` не используется, это не считается ошибкой;
- включать `ufw` только ради мониторинга не нужно.

Главный риск:

- если включить `ufw` без правила для SSH, можно потерять доступ к серверу.

Безопасный минимум перед включением `ufw`:

```bash
sudo ufw allow 22/tcp
sudo ufw allow OpenSSH
sudo ufw status
```

Если вы не уверены в текущей сетевой схеме сервера, лучше сначала:

- оставить `ufw` как есть;
- убедиться, что SSH стабильно доступен;
- проверить VPN, NAT, маршрутизацию и только потом менять firewall.

## Установка и первая проверка

1. Откройте страницу мониторинга нужного сервера.
2. Если утилиты не установлены, приложение покажет предупреждение и команды установки.
3. После установки дождитесь загрузки карточек мониторинга.
4. Проверьте, что сервер отвечает по SSH без ошибок таймаута и авторизации.

Быстрые проверки на самом сервере:

```bash
systemctl --failed --no-legend --plain
ss -tuln
sudo journalctl --since "24 hours ago" --no-pager | tail -50
sudo fail2ban-client status sshd
```

## Как читать данные правильно

### Firewall

Если в UI:

- `backend = none` и есть listening ports, это нормально: сервисы слушают порты, но явный firewall backend не обнаружен;
- `backend = ufw`, но `status = inactive`, это тоже допустимо;
- `backend = firewalld` или `nftables`, это не ошибка, а просто другая конфигурация сервера.

### Security

Если много `ssh_failures_24h`, это не всегда инцидент.

Смотрите вместе:

- топ атакующих IP;
- значение `fail2ban_banned`;
- число `failed_services`;
- число `error_events_24h`.

Именно комбинация этих сигналов лучше отражает реальную проблему, чем одна метрика.

### New open ports

Метрика строится относительно baseline-файла:

- `/var/tmp/open_ports_baseline.txt`

Первый запуск обычно просто создает baseline. После этого новые listening ports считаются относительно уже сохраненного состояния.

---

## Практические рекомендации для админа

- Не считайте "firewall выключен" автоматически проблемой. Сначала смотрите на архитектуру сервера.
- Оценивайте реальные listening ports, а не только список правил firewall.
- Для SSH-защиты полезнее корректно настроенный `fail2ban`, чем случайно включенный `ufw`.
- Если появились новые порты, сверяйте их с реально установленными сервисами и деплоем.
- Если растет `error_events_24h`, сразу проверяйте `journalctl`, а не только UI.
- Если есть `failed_services > 0`, сначала исправляйте systemd unit'ы, потом уже разбирайте вторичные симптомы.

---

## Ограничения текущей реализации

- security updates сейчас ориентированы на Debian/Ubuntu-подобные системы через `apt`;
- `ufw`-специфичные блокировки читаются только если реально есть `ufw`;
- список сервисов фиксированный и покрывает только типовые сервисы;
- история CPU/Memory хранится локально на сервере и не является полноценной TSDB;
- мониторинг не заменяет Prometheus/Grafana или специализированный NMS.

## Troubleshooting

### Не удается подключиться по SSH

Проверьте:

```bash
ssh user@server
```

И затем:

- правильный IP;
- правильный SSH port;
- корректный логин и пароль;
- не блокирует ли доступ firewall или хостинг-provider ACL.

### В UI пусто по security-событиям

Проверьте на сервере:

```bash
sudo journalctl --since "24 hours ago" --no-pager | tail -50
sudo grep "Failed password" /var/log/auth.log | tail -20
```

На части систем основным источником будет journald, на части - классический `auth.log`.

### Нет суточного трафика

Проверьте:

```bash
command -v vnstat
systemctl status vnstat
vnstat --json
```

### Нет списка listening ports

Проверьте:

```bash
ss -tulnH
netstat -tuln
```

Если `ss` недоступен, приложению поможет наличие `net-tools`.

### Много ошибок в journal

Начните с:

```bash
sudo journalctl --since "24 hours ago" -p err..alert --no-pager
systemctl --failed --no-legend --plain
```

## Для production

Этот мониторинг подходит для встроенного оперативного обзора сервера из UI приложения.

Для более зрелого production-стека имеет смысл дополнительно использовать:

- внешний uptime-check;
- централизованный сбор логов;
- резервный канал доступа к серверу;
- отдельную систему метрик и алертов.

Но даже без этого текущая реализация уже дает полезный минимум:

- сетевой обзор;
- SSH/security сигналы;
- видимость по firewall backend;
- контроль сервисов;
- базовую историю производительности.
