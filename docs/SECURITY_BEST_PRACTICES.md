# Рекомендации по безопасности для панели управления VPS

Этот документ описывает текущую модель безопасности приложения и даёт рекомендации по её улучшению в будущем.

## 1. Текущая модель безопасности

Приложение использует многоуровневый подход к защите данных, где разные типы информации защищаются по-разному в зависимости от их чувствительности.

### Хранение данных

*   **Конфиденциальные данные:** Вся информация о серверах хранится в файле `data/servers.json`.
*   **Ключ шифрования:** Главный ключ для шифрования и расшифровки паролей (`SECRET_KEY`) находится в файле `.env` в корне проекта.

**КРИТИЧЕСКИ ВАЖНО:** Файл `.env` **никогда** не должен попадать в систему контроля версий (Git) или публиковаться. Он содержит ключ ко всем вашим зашифрованным данным.

### Уровни безопасности по типам данных

| Тип данных | Метод защиты | Формат хранения | Уровень безопасности | Пояснение |
| :--- | :--- | :--- | :--- | :--- |
| **Пароли** (SSH, Панель, Хостер) | **Симметричное шифрование** (Fernet) | Зашифрованный | **Высокий** | Пароли никогда не хранятся в открытом виде. Они шифруются с помощью `SECRET_KEY` перед сохранением в JSON. |
| **Логины** (Панель, Хостер) | **Симметричное шифрование** (Fernet) | Зашифрованный | **Высокий** | Логины также шифруются для комплексной защиты учётных данных. |
| **URL-адреса** (Панель, Хостер) | **Маскировка (сокрытие)** | Открытый текст | **Низкий** | URL хранятся в открытом виде, но на главной странице их пути скрываются для защиты от случайного взгляда. |
| **Прочие данные** (IP, заметки и т.д.) | **Отсутствует** | Открытый текст | **Низкий** | Общая информация о серверах хранится как есть, в открытом виде. |

## 2. Будущие улучшения и лучшие практики

Текущая модель хорошо подходит для личного использования, но для повышения безопасности и масштабируемости проекта можно применить несколько лучших практик.

### Рекомендация №1: Переход от JSON к базе данных (например, SQLite)

Это самое значительное улучшение, которое можно сделать.

*   **Почему это важно?**
    *   **Масштабируемость:** Базы данных (БД) гораздо лучше справляются с большими объёмами данных и одновременными запросами.
    *   **Надёжность:** БД следят за целостностью данных и их типами.
    *   **Безопасность:** В БД встроены мощные механизмы безопасности, которые упрощают управление доступом и предотвращают ошибки при одновременной записи.
    *   **Производительность:** БД оптимизированы для быстрого поиска и обновления информации.
*   **Как применить:**
    1.  Выберите простую БД, например **SQLite**, которая хранит все данные в одном файле и не требует отдельного сервера.
    2.  Используйте библиотеку **SQLAlchemy**. Она позволяет работать с данными как с Python-объектами, что делает код чище и безопаснее.
    3.  Напишите небольшой скрипт для миграции: он один раз прочитает данные из `data/servers.json` и перенесёт их в новую базу данных.

### Рекомендация №2: Использовать переменные окружения для всей конфигурации

Вместо `config.json` лучше использовать `.env` для всех параметров конфигурации.

*   **Почему это важно?** Это стандартная практика, позволяющая отделить код от конфигурации, которая может содержать чувствительные данные.
*   **Как применить:**
    1.  Перенесите все настройки, например, `app_info` из `config.json` в файл `.env`.
    2.  Создайте файл `.env.example`, который будет содержать список всех нужных переменных, но с пустыми или примерными значениями. Этот файл можно безопасно добавлять в Git, и он будет служить шаблоном для настройки проекта.

### Рекомендация №3: Внедрить журнал аудита (логирование)

Отслеживайте важные события в приложении.

*   **Почему это важно?** Журнал аудита помогает понять, кто, что и когда сделал. Это бесценно для анализа безопасности и отладки.
*   **Как применить:**
    1.  Создайте новый механизм логирования (например, в файл `logs/audit.log` или в отдельную таблицу БД).
    2.  Записывайте такие события, как: "Пользователь X добавил сервер Y", "Пользователь X просмотрел пароль от сервера Z" и т.д.

### Рекомендация №4: Подготовка к реальному использованию (Production) с HTTPS

Никогда не используйте встроенный сервер Flask для реальной работы.

*   **Почему это важно?** Он небезопасен и не рассчитан на нагрузки. Весь трафик, включая пароли, которые вы просматриваете, передаётся в незашифрованном виде (HTTP).
*   **Как применить:**
    1.  Используйте профессиональный WSGI-сервер, например **Gunicorn**.
    2.  Настройте его для работы через обратный прокси-сервер **Nginx**.
    3.  С помощью **Let's Encrypt** получите бесплатный SSL/TLS-сертификат для Nginx, чтобы включить **HTTPS**. Это зашифрует весь трафик между браузером и вашим сервером.
